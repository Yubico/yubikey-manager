== Working with the code

=== Repository structure

This repository is a mixed Rust/Python workspace:

[cols="1,2"]
|===
| Path | Description

| `crates/yubikit`
| Core Rust library — protocol implementations for all YubiKey applications

| `crates/yubikit-pyo3`
| PyO3 bindings that expose the Rust library to Python

| `crates/ykman`
| Rust library implementing higher-level ykman operations

| `crates/ykman-cli`
| The `ykman` command-line executable

| `crates/ykman-svc`
| Background service for elevated YubiKey access (Windows)

| `packages/yubikit`
| Python package wrapping the native Rust extension

| `packages/ykman`
| Python wrapper for the `ykman` command-line executable

|===

=== Install dependencies

This project uses https://rustup.rs/[rustup] for the Rust toolchain. The
Python parts use https://docs.astral.sh/uv/[uv], which can be skipped when
only developing the Rust parts. Note that changes in the `yubikit` crate often
require modifications to the Python wrapper as well.

==== Linux (Debian-based distributions)

    $ sudo apt install pcscd libpcsclite-dev libudev-dev

==== Linux (RPM-based distributions)

    $ sudo dnf install pcsc-lite-devel systemd-devel

==== macOS

No additional system packages required.

==== Windows

No additional system packages required.

=== Setting up the development environment

Clone the repository and set up the Python environment:

    $ git clone https://github.com/Yubico/yubikey-manager.git
    $ cd yubikey-manager
    $ uv sync

Build the native Rust extension and install it into the virtual environment:

    $ cd crates/yubikit-pyo3
    $ maturin develop
    $ cd ../..

After making changes to the Rust code, re-run `maturin develop` to rebuild
and reinstall the native extension before testing.

NOTE: Use `.venv/bin/python3 -m pytest` (or activate the venv) rather than
`uv run pytest` when testing after `maturin develop`.  The `uv run` command
synchronises the environment before running, which can overwrite the
maturin-installed native extension with a cached wheel.


=== Building the Rust crates and executables

To build the project use cargo build:

    $ cargo build

For a release build:

    $ cargo build --release

=== Building the ykman executable

To build a debug binary:

    $ cargo build -p ykman-cli

The binary is written to `target/debug/ykman`.

For a release build:

    $ cargo build --release -p ykman-cli

The binary is written to `target/release/ykman`.

The `hardware` feature (enabled by default) links against libpcsclite and
hidapi to communicate directly with connected devices. Build without it to
produce a binary that can only communicate via the `ykman-svc` service:

    $ cargo build -p ykman-cli --no-default-features

=== Building the Python wrapper

To build the Python wrapper, use `uv`:

    $ uv sync

Because uv caches wheels, any changes to the native layers may require you to
force a rebuild of the yubikit package:

    $ uv clean && uv pip uninstall yubikit
    $ uv sync

=== Code style & security

https://docs.astral.sh/ruff/[ruff] handles Python style and security linting.
https://mypy-lang.org[mypy], https://github.com/microsoft/pyright[pyright], and
https://github.com/astral-sh/ty[ty] are used for static type checking.
Rust code uses `cargo fmt` and `cargo clippy`.
All checks are wired up via a http://pre-commit.com/[pre-commit] hook:

    $ uv tool install pre-commit
    $ pre-commit install

To run all checks manually:

    $ pre-commit run --all

=== Unit tests

These tests do not require a YubiKey to be connected.

Rust unit tests:

    $ cargo test --lib

Python unit tests:

    $ uv run pytest

=== Rust device tests

WARNING: ONLY run these on dedicated developer keys, as it will permanently
delete data on the device!

The test suite automatically detects whether the device is connected over USB or NFC.
Tests that require a physical USB connection (e.g., OTP/FIDO HID) are automatically
skipped when the device is connected over NFC.

    $ YUBIKEY_SERIAL=123456 cargo test -p yubikit --test device_tests -- --test-threads=1 --no-capture

For devices without a serial number, use `YUBIKEY_NO_SERIAL=1` instead:

    $ YUBIKEY_NO_SERIAL=1 cargo test -p yubikit --test device_tests -- --test-threads=1  --no-capture

The same command works whether the device is connected via USB or via an NFC reader.

You can target specific tests by specifying the test name after the command,
eg. append "fido" to only run the FIDO tests.

NOTE: --test-threads 1 is required as the device tests interact with a connected
YubiKey and cannot be run in parallel. --no-capture displays output showing which
tests were skipped, and why.

=== Python device tests

WARNING: ONLY run these on dedicated developer keys, as it will permanently
delete data on the device!

Specify the serial number of the YubiKey to test with (shown by `ykman list`):

    $ uv run pytest --device 123456

This works for both USB and NFC-connected devices. The test suite automatically
skips tests that are not applicable to the attached device.

=== Generating documentation

==== Rust (cargo doc)

Generate and open API documentation for the `yubikit` crate:

    $ cargo doc -p yubikit --open

To include documentation for all workspace crates:

    $ cargo doc --workspace --open

The generated HTML is written to `target/doc/`.

==== Python (Sphinx)

Generate HTML API documentation for the Python packages:

    $ uv run sphinx-build docs docs/_build/html

Open `docs/_build/html/index.html` in a browser to view the result.
